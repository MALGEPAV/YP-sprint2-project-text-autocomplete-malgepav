### __Папка data:__
Все файлы кроме folder_description.txt создаются командами из solution.ipynb и не попадают в github
* tweets.txt - исходный файл с твитами
* cleaned_tweets.csv - "очищенные"  твиты, при создании этого файла можно указать какую часть всех твитов использовать (на всех слишком долгое обучение и валидация)
* train_tweets.csv - "тренировочные твиты"
* val_tweets.csv, test_tweets.csv - валидационный и тестовый датасеты в виде пар: начало твита и его окончание, брались твиты с не менее 4 словами, так как при валидации предсказываем последнюю четверть твита по первым трем четвертям
* folder_description.txt - добавлен просто чтоб на github залилась папка, пустая папка не заливается

* Тут сохраняются результаты каждой "эпохи" обучения. Файл folder_description.txt просто чтоб папка залилась в github, так как пустая папка не пушится. В github не пушатся чекпойнты, так как они тяжелые

### __Папка src:__
* data_utils.py - содержит функции для закачки файла с твитами, очистки их и записи в cleaned_tweets.csv, создания тренировочного, валидационного и тестового csv-файлов 
* next_token_dataset.py - содержит класс тренировочного датасета в который попадает каждый твит из не менее 4 слов и разбивается на несколько тренировочных примеров. Например твит "i love books" породит примеры "i" -> "love", "i love" -> "books".  Также содержит функцию для создания этого датасета по пути к csv файлу и функцию, возвращающую специальную функцию collate_fn для соответсвующего даталоудера,в которой несколько групп обучающих примеров, отвечающих одному твиту объединяются в общие тензоры.
* seq_generation_dataset.py - содержит функции аналогичные функциям файла next_token_dataset.py, но для валидационного датасета с парами начало-конец твита
* lstm_model.py - файл с классом lstm-модели.
Модель имеет метод forward для генерации одного следующего токена и метод generate, который по началу предложения генерирует его окончание, при заданном максимальном количестве новых токенов.
К сожалению, метод generate работает с одним предложением.
Реализация генерации окончаний для батча показалась мне слишком сложной, учитывая, что нужно отбрасывать предложения в которых сгенерился eos_token, и вообще генерировать для каждого предложения разное количество токенов, так как предсказываем последнюю четверть.
* lstm_train.py - содержит функцию тренировки одной эпохи, которая возвращает средний суммарный loss по батчу по всему даталоудеру
* eval_lstm.py - содержит функцию, возвращающую метрики rouge1 и rouge2, описывающие эффективность модели в генерации окончаний предложений на валидационном датасете
* При генерации окончания max_new_tokens равен трети длины начала, то есть предсказываем последнюю четверть твита.
* Сначала входные последовательности input_ids для моделей я получал при помощи метода decode у токенизатора, который дает только input_ids, но оказалось,что тогда модель gpt2 начинает постоянно выдавать предупреждение что не задан attention mask и еще что явно не указан pad_token_id и она его приравняет к eos_token_id. Эти предупреждения засоряли вывод сгенерированных предложений. Поэтому я входные последовательности стал получать по схеме 
tokenizer(string). Тогда получаем и attention mask. Плюс дополнительно пришлось писать pad_token_id=tokenizer.eos_token_id в методе generate у gpt2. Чтобы при этом можно было применять одну и ту же функцию и для оценки своей модели и gpt2, в методе generate у своей модели пришлось добавить **kwargs, чтобы ее не смущали всякие attention mask и прочее.

### __Модель__

 * к выходам rnn слоя применяется mean_pooling, показалось что не повредит учитывать выходы всех ячеек
 * чтобы при mean_pooling не учитывать выходы, отвечающие паддингу, применяется запаковка-распакова при помощи функций 
pack_padded_sequence и pad_packed_sequence
